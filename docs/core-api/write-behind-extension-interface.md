# 쓰기 지연(write-behind) 확장 인터페이스(후보)

## 목적
- 이벤트 처리 경로에서 write-behind(`wb_worker`) 확장 지점을 위한 설계 목표를 정의합니다.
- 향후 transform/filter 로직을 허용하면서도 신뢰성과 관측성 보장을 유지합니다.

## 훅 표면 후보
- **쓰기 전 변환 훅(Pre-write transform hook)**: 디코딩된 스트림 이벤트를 입력받아 변환된 이벤트 payload를 반환
- **필터 훅(Filter hook)**: DB 쓰기 전에 사유 코드와 함께 keep/drop 결정을 반환
- **실패 분류 훅(Failure classification hook)**: 쓰기/검증 실패를 retry, DLQ, drop 정책으로 매핑

## 실패 의미 규칙
- 훅 예외/실패 시 기본 동작은 보수적 경로를 따르며, 묵시적 성공 처리하지 않습니다.
- 분류 훅이 실패하면 이벤트는 기존 오류 처리 정책(`WB_ACK_ON_ERROR`, DLQ/dead stream 설정)을 따릅니다.
- drop 결정은 감사 가능성을 위해 사유 태그가 포함된 카운터를 반드시 남겨야 합니다.

## 관측성 계약
- 향후 확장 경로에서 필요한 메트릭:
  - 훅 호출 총량
  - 훅 오류 총량
  - 변환/필터 결정 총량
  - fallback 경로 총량
- 메트릭 이름은 Prometheus 친화성을 유지하고 현재 `wb_*` 네이밍 스타일과 일관되어야 합니다.

## 호환성 표면
- Stable 후보 인터페이스는 고정 이벤트 DTO 버전과 훅 결과 스키마를 정의해야 합니다.
- 훅 계약의 파괴적 변경에는 마이그레이션 노트와 호환성 분류 갱신이 필요합니다.

## 이번 단계 비목표
- 런타임 훅 실행 엔진 구현 없음
- 기존 write-behind 영속화 흐름 변경 없음
